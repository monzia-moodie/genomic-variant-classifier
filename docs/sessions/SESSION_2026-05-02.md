### 3. LOVD data on disk is structurally joinable

`data/external/lovd/lovd_all_variants.parquet` is healthy: 18,006 rows
across 10 genes (APC, BRCA1, BRCA2, MLH1, MSH2, MSH6, NF1, PTEN, RB1,

classification_raw="pathogenic", source_format="lovd_tab"`.

Diagnostic merge (`diag_lovd_join.py`) replicating the connector's exact
key-construction logic against `models/v1/clinvar_enriched.parquet`
(1,700,687 rows, 54,710 of which are in LOVD's 10-gene set) yields
**5,553 inner-join matches**. Both sides produce string-typed `_chrom`,
`_pos`, `_ref`, `_alt` keys with identical dtypes; no float→str
trailing-`.0` issue, no `chr` prefix mismatch, chromosome sets overlap
cleanly.

### 4. Cause is at a runtime join boundary, not in the call site or the data

The contradiction (5,553 matches in isolation, 0 in the trained matrix)
can be reconciled exactly two ways:

**Cause 1 — Downstream column overwrite.** The connector reports 5,553
matches at runtime, but a downstream step (`_engineer_features`, `_scale`,
or a fillna) overwrites `lovd_variant_class` with zeros before the
column reaches `X_train.parquet`.

**Cause 2 — Upstream coordinate transformation.** The connector reports
0 matches at runtime because one of the 14 prior `annotate_dataframe`
calls (steps 1–14, with step 14 being `protein.annotate_dataframe(df)`
from `ProteinStructurePipeline`) has transformed `chrom`/`pos`/`ref`/
`alt` in `df` before the LOVD step at line 738 sees it.

The log line at `real_data_prep.py:740–748` distinguishes them:

```python
logger.info(
    "Score annotation 15/16 (LOVD): %d variants with lovd_variant_class > 0.",
    int((df.get("lovd_variant_class", pd.Series([0] * len(df), index=df.index)) > 0).sum()),
)
```

Whatever integer this emitted during the run9_ready regen on 2026-04-30
is the answer:

- non-zero (≈5,500) → Cause 1, fix is in `_engineer_features`/`_scale`
- zero → Cause 2, fix is in one of the upstream connectors corrupting
  the join keys

Resolution deferred to R10-A (grep `outputs/run9_ready/regen.log` —
which `SESSION_2026-04-30.md` confirms was written — for the
`Score annotation 15/16 (LOVD)` line). Two-minute task.

### 7. Adjacent: "Weaker labels" claim grounded

The earlier-session claim that LOVD-API-derived labels would be weaker
than ClinVar tier-2/3 was challenged mid-session ("is this an
assumption?"). Re-grounded against two sources:

- **LOVD admin's 2026-04-01 second email**: "The clinical classification
  field still hasn't been fully standardized, as we decided to wait for
  ACMG v4, and as such, it's not included in the APIs yet." API exposes
  `effect_reported` (submitter) and `effect_concluded` (curator), with
  the latter "may be used heavily or not at all" depending on per-gene
  curator activity. `functionalProbablyAffected` "translates to" likely
  pathogenic — admin's word; functional, not clinical.
- **ACMG/AMP 2015 (Richards et al.)**: defines clinical classification
  by combining evidence across multiple categories (population
  frequency, computational, functional studies PS3/BS3, segregation, de
  novo, allelic, etc.). Functional evidence is one input, not the
  classification itself. Pipeline currently trains on ClinVar tier-2
  ("criteria provided, multiple submitters, no conflicts") clinical
  classifications — categorically different from a functional-
  translated-to-clinical label.

The earlier-session "30× more rows" figure was rhetorical and was
flagged as such mid-session. Actual LOVD-side row contribution per new
gene depends on per-gene curator activity and would have to be measured
from the downloaded `.txt` files.

## Process violations recorded this session

- **`PASTE_FULL_PATH_HERE` placeholder in copy-pasteable command.**
  Earlier in the session I issued a `python -c` command with that
  literal string as the parquet path, after the discovery command had
  returned 15 candidate `X_train.parquet` paths. Should have either
  run discovery first and issued a concrete-path command, or used a
  `$path = ...; python -c ...` two-step. Re-issued with the actual
  `outputs/run9_ready/splits/X_train.parquet` path on the next turn.

- **Bash heredoc syntax in PowerShell context.** Wrote
  `python - <<'PY' ... PY` for a 60-line diagnostic script. PowerShell
  parsed `<<` as a redirection operator and ran every subsequent line
  as PowerShell — 80+ cascade parser errors before the closing `PY`
  fell through to launch `py.exe` interactively. Standing rule #6 of
  the Windows-platform learnings already covers this ("PowerShell
  heredocs corrupt when content contains triple-quoted Python strings
  or literal commit messages — use standalone `.py` files instead").
  Re-issued as `@'...'@ | Out-File diag.py; python diag.py`.

- **Loose grep regex framed as decisive.** First search pattern was
  `annotate_dataframe|LOVDConnector`, which matched method definitions
  on every connector that defines such a method (clingen, dbnsfp,
  dbsnp, eve, hgmd, omim, phylop, revel, sift_polyphen, vep, esm2,
  lovd) and produced a 40-row table that was indistinguishable between
  "is X used" and "is X defined". Closed the response with "no
  ambiguity, four possible combinations, each with a definite
  implication" — overconfident framing of an under-constrained tool.
  Re-issued with tightened pattern matching instantiation, import,
  and attribute-access only.

- **Falsified hypothesis on float→str trailing-`.0`.** Pre-emptively
  diagnosed the silent-zero as a `pos` int64 vs float64 conversion
  bug. Verification command showed `pos` is int64 with clean integer
  string conversion. Hypothesis falsified cleanly. The correction was
  short (one paragraph) and proceeded directly to a more complete
  diagnostic that yielded the actual evidence. Recorded as a normal
  hypothesis-falsification cycle, not a rule violation, but worth
  noting that the rate of correct first-guesses on silent-zero causes
  is lower than I framed it.

Pattern across all four: confident framing on under-constrained
tooling. Recorded for future-self correction; the diagnostic-merge
pattern (replicate the production logic in isolation, see what it
produces, compare to production output) was decisive where the
cause-by-cause guessing was not.
