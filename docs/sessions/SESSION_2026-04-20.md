## Phase 1 — OOF bug diagnosis and fix

Read `src/models/variant_ensemble.py` lines 899-1164 (the full
`VariantEnsemble` class including `fit()`). Confirmed the hypothesis:
a botched earlier patch at lines 1156-1163 left a duplicate
`meta_learner.fit()` pair. The first call (line 1159) used `y_arr`
(full length N), crashed on `check_consistent_length` because
`oof_preds` has length 0.85 × N (derived from the calibration split
at line 1086). The second call one block below correctly used
`y_fit` but never ran because the first call always crashed first.

Fix: delete lines 1156-1159 (the stale block including the duplicate
`logger.info`), keep the `y_fit` call. Net change: `-7/+1` lines.

Verified with a 500-row synthetic probe calling `VariantEnsemble.fit`
with `skip_svm=True`, `skip_catboost=True`, `skip_mc_dropout=True`
and the two torch-backed neural models popped from `base_estimators`.
Result: `FIT SUCCEEDED. oof_predictions_.shape: (425, 6)`. 425 = 0.85
× 500 as predicted. Meta-learner fit completed. Blend weights
populated. `oof_predictions_` and `oof_model_names_` attributes
exposed correctly for Rule-5 artefacts downstream.

The probe also trained KAN as one of the 6 models — `_KAN_AVAILABLE`
is True locally because `pykan` is installed, and no skip flag was
set. This set up Phase 2.

## Phase 2 — KAN history reconstruction

Initial plan after Phase 1 was to commit the OOF fix, then proceed
with Patch v4.1/v4.2/v4.3 to add `--skip-kan` to the mainline and
ablation scripts. Before applying, user asked for thorough
investigation of *why* KAN had been marked "unconditionally removed"
and whether that decision remained applicable given Vast.ai GPU access.

The investigation ran a four-part grep over the repo:

1. `docs/` (markdown, plain text, JSON, YAML)
2. `logs/`
3. Source and config (`.py`, `.ipynb`, `.ps1`, `.sh`, `.bat`, `.cmd`,
   `.toml`, `.cfg`, `.ini`, `Dockerfile`)
4. `git log --all --grep='KAN|pykan' --regexp-ignore-case`

Key findings:

- **Commit cff925c (2026-03-26)** introduced
  `src/models/kan.py` with a three-backend fallback (pykan → efficient-kan
  → MLP). The model is always usable regardless of which library is
  installed.

- **Commit 2389ee2 (2026-04-04)** fixed the 17.9 GB OOM by adding a
  100K-sample stratified subsample gate in `KANClassifier._fit_pykan`.
  Commit message records peak RAM ~0.3 GB vs 17.9 GB. This happened
  15 days before the session.

- **Commit a0a732d (2026-04-05, Run 6 prep)** decided "no KAN" for
  Run 6 despite the same-day fix being shipped. Belt-and-braces
  paranoia. The hardcoded `ensemble.base_estimators.pop("kan", None)`
  in `scripts/run_phase2_eval.py:175` dates to this prep and has
  persisted through Runs 6, 7, 8.

- **Commit 671e48d (2026-04-09)** added `--skip-kan` to the Dockerfile
  trainer CMD at line 166. But `run_phase2_eval.py` did not accept
  that flag until today. Dockerfile trainer has been broken for 11
  days; nobody noticed because all training used startup scripts, not
  Docker.

- **Commit a95c9db (2026-04-09)** shipped `agent_layer/agents/version_monitor_agent.py`,
  which programmatically monitors pykan PyPI releases for OOM/memory
  fixes. Real code, not planning.

- **`docs/ROADMAP.md` lines 206-212** has an explicit KAN Re-enablement
  Checklist. Every item on that checklist maps cleanly to the session's
  planned work.

- **`docs/PHASE_3_ROADMAP.md` line 15** records the 17.9 GB fix
  with commit reference.

Conclusion: the "unconditionally removed" status was out of date. The
OOM fix had been in the codebase for 15 days. Vast.ai access made
even the fallback path (100K subsample) unnecessary for Run 9. The
correct move was to execute the ROADMAP re-enablement checklist
exactly as written.

## Phase 3 — Reversal decision

User instruction: "I want KAN included in Run 9's full ensemble
baseline AND as an ablation target (`no_kan`), not just 'don't delete
the code'."

Verified Dockerfile classification: CPU-only multi-stage. All three
stages (builder, api, trainer) use `python:3.11-slim-bookworm`. No
CUDA base image. GPU training goes through startup scripts, not
Docker. No Dockerfile change needed for Run 9 per se; the `--skip-kan`
in the trainer CMD becomes valid once the argparse flag exists.

Memory rules updated to reflect the reversal. Previously held
"unconditionally removed" status replaced with current reinstatement
decision and references to both the 2389ee2 fix and Vast.ai access.
