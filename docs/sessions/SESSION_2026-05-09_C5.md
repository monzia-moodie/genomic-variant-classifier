## Lessons learned

### 1. Apply-batch arithmetic-sanity rule (NEW STANDING RULE)

Every mechanical-rewrite batch must assert at Stage 1 (dry-run), before any file is written:

    expected_substitutions  =  baseline_lines  -  deliberate_skip_lines  -  incidental_lines  +  multi_match_extras
    actual_substitutions   ==  expected_substitutions

Each term, with its source and C5.3a v2 value:

| Term | Source | C5.3a v2 |
|------|--------|---------:|
| `baseline_lines` | Pre-flight `Select-String` count of stale-ref LINES across the allow-list (default `Select-String` reports one hit per line, ignoring multi-match-per-line). | 87 |
| `deliberate_skip_lines` | Total entries across all `SKIP_LINES[file]` sets in the helper — lines we want to preserve unchanged. | 4 |
| `incidental_lines` | Lines that match the broad discovery regex but are deliberately not matched by the (narrower) apply regex. Documented per case. | 1 (test_1kgp.py L409 bare `src/`) |
| `multi_match_extras` | For each non-skipped line where the apply regex matches more than once, add `(matches_per_line - 1)`. Multi-match lines contribute extra substitutions beyond their line-level count. | 1 (Dockerfile L10: 2 matches on 1 line, +1) |
| `actual_substitutions` | The Python helper's final `Combined replacements: N` line, parsed from dry-run stdout. | 83 |

So C5.3a v2: `83 == 87 - 4 - 1 + 1` — the assertion holds, the batch proceeds.

C5.3a v1 had the same expected = 83, but its dry-run silently reported 78 (5 substitutions undershot due to the required-trailing-slash regex). With no Stage 1 assertion, the mismatch went unnoticed until Stage 3 post-apply verification — after 51 files had already been partially modified. With the Stage 1 check in v2, the same class of bug aborts at dry-run time, before the working tree is touched.

Notes:
- `variant_ensemble.py L20` also has 2 matches on a single line (`src/models/ensemble.py + src/models/variant_ensemble.py`), but L20 is in `SKIP_LINES` — so it contributes zero to `actual_substitutions` and zero to `multi_match_extras`. Multi-match-per-line only counts on lines that are actually touched.
- For a future apply where the audit is purely single-match-per-line, `multi_match_extras` is 0 and the formula collapses to the simpler `expected = baseline_lines - skip_lines - incidental_lines`.

Codify this in every future apply helper template.

### 2. Path-style regex: word-boundary lookahead > required trailing slash

Robust form:

    re.compile(r"(?<![A-Za-z0-9_])src/(api|data|...)(?![A-Za-z0-9_])")

Fragile form (v1):

    re.compile(r"(?<![A-Za-z0-9_])src/(api|data|...)/")

The lookahead form catches references at end-of-line, before space/comma/dot/paren, while still rejecting partial matches (e.g., `src/apify` won't be read as `src/api` + `fy` because `f` is alphanumeric).

### 3. Recovery enforced by pre-flight, not optional

Apply batches' pre-flight rejects dirty working trees. After a Stage 3 fail, `git checkout -- .` is mandatory before retry. This prevents v1-then-v2-on-dirty-state confusion (where v2 would only do new substitutions on top of v1's partial work, producing a misleading dry-run total that doesn't match a clean-state baseline).

### 4. Per-line skip lists for mixed-history files

Files like `variant_ensemble.py` mix forward-looking module references (rewrite) with historical Phase-1 commentary (preserve). A `SKIP_LINES` dict in the helper cleanly distinguishes them; the helper iterates lines, applying regex only to non-skipped lines.

### 5. Substitutions ≠ line-level diff

Helper output (`subn` return value) and git diff stat differ when a single line has multiple substitutions. C5.3a's commit shows `+82/-82` (lines changed) while the helper reported 83 substitutions — Dockerfile L10 had 2 substitutions on a single line (the same multi-match line that drives `multi_match_extras = 1` in Lesson 1).
